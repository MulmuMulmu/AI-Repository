프로젝트 개요
물무물무는 1인 가구 사용자가 구매한 식재료를 기록하고 소비하며, 남은 재료를 나눔까지 연결하는 과정을 하나의 흐름으로 묶기 위해 기획한 생활형 AI 서비스이다.
1장에서는 서비스가 해결하려는 문제와 개발 범위를 먼저 설명하고, 이후 장의 기능 설계와 구현 계획을 읽는 기준이 되는 일정과 역할 구조를 함께 제시한다.
개발배경
■ 문제 배경
국내 1인 가구의 비중과 절대 가구 수는 꾸준히 증가하고 있으며, 식재료 구매와 보관, 소비의 기준도 가족 단위에서 개인 단위 생활 관리로 빠르게 이동하고 있다.
하지만 1인 가구의 식생활은 소포장 구매의 한계, 불규칙한 식사 패턴, 장보기 이후의 기억 의존형 관리 때문에 재료가 남기 쉬운 구조를 갖는다. 따라서 이 문제는 단순한 소비 습관이 아니라 생활 관리 체계의 부재에서 비롯된다고 볼 수 있다.
그림 1.1 1인 가구 증가와 식품 폐기 문제의 확대
위 통계는 서비스의 잠재 사용자가 확대되고 있다는 사실만 보여주는 것이 아니라, 재고 관리 실패가 일부 사용자의 특수한 문제가 아니라 일상적인 생활 문제로 확대되고 있음을 함께 보여준다.
■ 식품 폐기 문제의 규모
사용하지 못한 식재료의 폐기는 개인 차원의 손실에서 끝나지 않는다. 개봉하지 못한 식품, 소비 시점을 놓친 재료, 불필요한 중복 구매는 가계 지출 증가와 음식물 쓰레기 처리 비용 증가로 이어진다.
그림 1.2 식품 폐기로 인한 사회경제적 손실
그림 1.3 식품 폐기 증가와 환경 비용 문제
기사 자료가 보여주듯 식품 폐기 문제는 생활비와 환경 비용을 동시에 키우는 구조적 문제다. 따라서 물무물무의 개발목표는 단순한 기록 보조 기능을 만드는 것이 아니라, 사용자가 버리기 전에 행동을 바꾸도록 돕는 서비스 구조를 설계하는 데 있다.
■ 기존 서비스 분석과 벤치마킹
기존 서비스는 각각 유의미한 강점을 갖고 있으나, 재고 관리에서 소비와 나눔까지 이어지는 전체 흐름을 하나의 경험으로 연결하지는 못한다. 따라서 벤치마킹은 기능 나열이 아니라 흐름이 끊기는 지점을 파악하는 과정으로 읽어야 한다.
1) 당근마켓
그림 1.4 당근마켓 벤치마킹
당근마켓은 지역 인증과 채팅 기반 거래 경험이 강점이며, 실제 사용자 간 교환과 커뮤니케이션을 자연스럽게 만든다. 그러나 식재료의 상태, 유통기한, 소비 우선순위를 서비스가 직접 다루지 않기 때문에 사용자는 거래 이전의 관리 판단을 스스로 수행해야 한다.
2) 유통기한 언제지
그림 1.5 유통기한 언제지 벤치마킹
유통기한 관리 앱은 재고를 기록하고 만료 시점을 확인하는 기능에는 효과적이다. 반면 기록 이후에 무엇을 먼저 소비할지, 어떤 재료를 레시피와 연결할지, 남는 재료를 어떻게 나눌지까지 이어지는 행동 전환 구조는 상대적으로 약하다.
3) 만개의 레시피
그림 1.6 만개의레시피 벤치마킹
레시피 서비스는 풍부한 요리 콘텐츠를 제공하지만 사용자의 실제 냉장고 상태와 직접 연결되지 않으면 탐색이 소비 행동으로 이어지지 못할 수 있다. 물무물무는 이 세 서비스의 강점을 참고하되, 입력 이후의 다음 행동을 서비스 안에서 이어지게 만드는 것을 목표로 한다.
개발목표와 범위
■ 물무물무 개발목표
그림 1.7 물무물무 구현 대상 시스템과 개발 범위
물무물무의 1차 개발목표는 영수증 OCR을 통해 식재료 입력 비용을 낮추고, 재고 조회, 임박 알림, 레시피 추천, 지역 기반 나눔을 하나의 서비스 흐름으로 연결하는  것이다.
즉 사용자가 재료를 기록한 뒤 다시 앱 밖에서 판단하지 않도록, 서비스 내부에서 소비와 공유 행동까지 이어지게 하는 것이 이번 프로젝트의 핵심 개발 방향이다. 이 목표는 입력 보조, 재고 관리, 소비 추천, 나눔 연결이라는 네 축으로 구체화된다.
■ 포함 범위와 제외 범위
이번 개발 범위에는 사용자 앱, 관리자 웹, 서비스 API, OCR·추천 AI 서버, 재고·추천·나눔의 핵심 시나리오가 포함된다. 운영 관점에서는 검수, 신고 처리, 기능 플래그, 파일럿 대상 관리 기능까지 함께 다룬다.
반면 결제 연동, 배송형 물류, 오프라인 정산, 점포 재고 자동 연동, 공동구매와 같은 확장 기능은 이번 졸업작품 범위에서 제외한다. 이는 사용자 핵심 행동을 우선 검증하고, 후속 확장 범위를 명확히 구분하기 위한 결정이다.
개발일정/산출물
■ 단계별 추진 계획
개발 일정은 분석과 설계, 구현, 통합 테스트, 발표 준비의 단계로 구분되며, 각 단계의 산출물이 다음 단계의 입력이 되도록 계획하였다.
그림 1.8 물무물무 개발일정
초기 단계에서는 문제 정의와 요구사항을 정리하고, 이후 설계서와 구현 계획을 기준으로 기능 단위를 병렬 개발한다. 테스트와 코드 검증 단계에서는 화면, API, AI 기능이 하나의 사용자 흐름으로 이어지는지를 함께 확인한다.
■ 단계별 산출물
주요 산출물은 제안서, 설계서, 사용자 앱 시연본, 관리자 웹 시연본, OCR·추천 기능 모듈, 테스트 결과, 최종 발표 자료와 최종 보고서로 정의한다.
각 산출물은 단순 제출 자료가 아니라 다음 단계의 기준 문서로 활용된다. 기능 범위나 일정이 바뀌는 경우 관련 산출물도 함께 갱신해 설계와 구현의 차이가 커지지 않도록 관리한다.
개발조직/역할
■ 조직 운영 원칙
개발 조직은 팀장, 백엔드·인프라, 앱·웹 프론트엔드가 각자의 책임 구간을 갖되, 하나의 기능을 화면·API·데이터·AI 규칙 단위로 함께 검토하는 구조로 운영한다.
특히 영수증 등록, 재고 조회, 추천, 나눔처럼 여러 계층을 동시에 거치는 기능은 설계 문서와 구현 결과를 함께 확인해 책임 경계가 흐려지지 않도록 관리한다.
■ 역할 분담
아래 표는 팀원별 책임 범위와 핵심 개발 기능을 정리한 것이다. 이후 장에서 설명하는 세부 기능과 구현 범위를 읽을 때 기준표로 활용한다.

■ 협업 방식
협업은 사용자 시나리오 단위로 진행하며, 화면 설계, API 계약, 데이터 구조, 운영 규칙이 같은 기준 문서 위에서 함께 갱신되도록 관리한다.

요구사항 정의
본 장에서는 물무물무 서비스가 제공해야 할 핵심 기능과 이를 뒷받침하는 데이터, AI, 운영 요구사항을 정리한다.
서비스 개요와 구성요소를 먼저 설명한 뒤, AI 서비스 유스케이스와 데이터분석 요구를 정의하고, 사용자·관리자·AI Server의 역할별 기능을 표와 짧은 설명 블록으로 정리한다.
서비스 개요
■ 서비스 목적
물무물무는 영수증 OCR을 통해 재고 입력 부담을 줄이고, 사용자가 현재 보유한 식재료를 기준으로 소비, 추천, 나눔까지 이어지는 행동 흐름을 하나의 서비스 안에서 경험하도록 설계한 생활형 서비스이다.
따라서 서비스 개요는 단순한 화면 목록이 아니라 사용자의 입력, 서비스 처리, AI 보조 처리, 운영 개입이 어떤 순서로 연결되는지를 설명하는 기준으로 읽어야 한다.
그림 2.1 물무물무 서비스 개요 구성도
■ 사용자 가치
사용자는 재고를 기록하기 위해 별도의 수작업을 반복하지 않고, 영수증 업로드 이후 재고 등록, 임박 알림, 레시피 추천, 지역 기반 나눔으로 자연스럽게 이동할 수 있어야 한다.
관리자는 검수와 신고 처리, 공지와 파일럿 운영을 하나의 운영 흐름 안에서 다뤄야 하며, 서비스 API와 AI 서버는 이러한 사용자 흐름을 안정적으로 지원하는 구조로 분리되어야 한다.
서비스 구성요소 정의
■ 주요 구성요소
물무물무는 사용자 앱, 관리자 웹, 서비스 API, OCR·정규화·추천 AI, 데이터 저장소와 외부 공개 레시피 코퍼스로 구성된다. 각 구성요소는 독립적으로 보이지만 실제 서비스에서는 하나의 사용자 시나리오를 기준으로 결합되어 동작한다.
그림 2.2 서비스 구성요소 구조도
사용자 앱은 영수증 업로드, 재고 조회, 추천 확인, 나눔 등록과 채팅을 담당한다. 관리자 웹은 OCR 검수, 신고 처리, 공지 관리, 파일럿 대상 운영을 담당한다.
서비스 API는 계정, 재고, 알림, 추천, 나눔, 운영 데이터를 관리하는 중심 계층이며, AI 서버는 OCR 추출과 정규화, 위험도 계산, 레시피 추천과 같은 비정형 판단 기능을 담당한다.
데이터 저장소는 재고, 추천, 나눔, 알림, 운영 로그를 안정적으로 적재하고, 외부 공개 레시피 코퍼스는 추천 품질을 보강하는 기초 데이터로 사용된다.
AI서비스 유스케이스 모델
AI 서비스는 사용자 앱과 관리자 운영이 만나는 지점에서 입력 자동화와 판단 보조를 담당한다. 따라서 AI서비스 유스케이스 모델은 단순한 내부 처리 흐름이 아니라 사용자 요청과 관리자 피드백이 어떻게 AI에 반영되는지를 함께 보여준다.
그림 2.3 AI서비스 유스케이스 모델
■ AI 서비스의 역할
AI 서비스의 핵심 역할은 영수증 이미지에서 품목을 추출하고, 품목명을 표준화하며, 유통기한과 임박도를 해석하고, 보유 재료를 기반으로 레시피 추천 후보를 생성하는 것이다.
이때 AI 결과는 사용자에게 즉시 확정값으로 노출되지 않고, 신뢰도와 검수 필요 여부를 함께 제시하는 보조 정보로 사용된다.
■ 사용자·관리자와의 상호작용
사용자는 OCR 결과를 수정하거나 재고를 직접 보완할 수 있어야 하며, 관리자 운영자는 검수와 피드백을 통해 정규화 사전과 예외 규칙을 개선할 수 있어야 한다.
즉 AI 서비스는 자동화 모듈이면서 동시에 사용자 입력과 관리자 운영 학습이 반복적으로 축적되는 구조를 가져야 한다.
데이터분석의 요구사항
■ 수집 데이터 범위
데이터분석의 요구사항은 단순히 많은 데이터를 확보하는 것이 아니라 OCR 추출, 재고 해석, 추천 생성, 나눔 전환을 평가할 수 있는 구조를 함께 확보하는 것이다.
영수증 데이터는 이미지 원본, OCR 텍스트, 품목명, 수량, 단위, 날짜 후보, OCR 신뢰도와 같은 기본 필드를 포함해야 하며, 잘림 여부와 촬영 품질 메타데이터도 함께 기록해야 한다.
레시피 데이터는 제목과 재료 리스트뿐 아니라 조리시간, 난이도, 대체재, 계절성, 남는 재료 처리 여부를 포함해야 추천 품질을 평가할 수 있다.
나눔 데이터는 게시 시간, 반경, 품목 상태, 채팅 시작 여부, 수령 완료 여부, 신고 발생 여부를 포함해 실제 전환 품질을 측정할 수 있도록 수집해야 한다.
■ 정제 및 품질 기준
정답셋은 품목명 정답만 저장하는 방식이 아니라 수정 전후 값, 실패 유형, 반려 사유, 규칙 반영 여부를 함께 남겨야 개선 근거를 잃지 않는다.
품목 정규화는 브랜드 제거, 동의어 통합, 단위 표준화, 카테고리 일치 여부를 기준으로 관리하며, 개인정보와 위치정보는 최소 수집 원칙에 따라 분리 저장한다.
결국 데이터분석의 요구사항은 수집량보다 구조와 재현성을 우선하는 방향으로 정의되어야 하며, 서비스 가치와 직접 연결되지 않는 필드는 배제하는 것이 원칙이다.

■ 기능별 간단 설명
1) 로그인
사용자는 카카오 소셜 로그인과 일반 로그인을 모두 사용할 수 있어야 한다.
일반 로그인은 아이디와 비밀번호를 입력받고, 로그인 이후에는 토큰 갱신과 로그아웃이 자연스럽게 이어져야 한다.
2) 회원가입과 선호 정보 등록
회원가입은 소셜 로그인 기반 가입과 일반 로그인 기반 가입으로 구분하며, 일반 가입 시에는 아이디, 비밀번호, 이름을 입력받아야 한다.
회원 가입 시에는 알러지, 선호 식재료, 기피 식재료를 함께 등록해 추천과 나눔 정책의 초기값으로 사용해야 한다.
3) 위치 설정과 영수증 등록
나눔 기능을 위해 GPS 기반 동네 인증을 수행하고, 지도 API를 이용해 주변 나눔 지역을 동 단위까지 표시해야 한다.
사용자는 카메라 촬영 및 이미지 업로드를 통해 영수증을 등록하고, OCR 인식 오류에 대비해 품목·수량·구매일을 수동 등록할 수 있어야 한다.
4) 재고 표시와 추천
등록된 식재료는 채소, 육류, 가공식품 등 아이콘 기반 카테고리로 표시하고, 날짜와 이름 기준 오름차순·내림차순 정렬을 지원해야 한다.
소비기한 계산은 영수증 OCR 결과를 GPT-4o-mini에 전달해 백그라운드로 수행하고, 결과는 화면에서 즉시 확인할 수 있어야 한다.
5) 레시피, 나눔, 안전 정책
레시피 추천은 없는 재료를 최대 50%까지 허용하는 방식과 보유 재료만 사용하는 방식을 모두 제공하며, 부족 재료는 빨간색 글씨로 구분하고 상세 화면에는 조리순서·난이도·예상 조리 시간을 제공해야 한다.
나눔 게시글 작성·조회·수정·삭제, 1:1 채팅, 나눔글 신고, 거래 완료 후 내 식재료 자동 삭제와 상대방 자동 등록, 개봉된 반찬·생고기·생선·조리된 음식 필터링, 유통기한 임박 알림까지 하나의 나눔 안전 흐름으로 연결되어야 한다.

■ 기능별 간단 설명
1) 관리자 웹 페이지와 로그인
관리자는 관리자 웹 페이지에 접속해 운영 대시보드를 조회할 수 있어야 하며, 일반 로그인으로 아이디와 비밀번호를 입력해 인증을 수행해야 한다.
권한 수준에 따라 접근 가능한 메뉴, 데이터 수정 범위, 계정 보안 설정을 구분하고 운영자 계정 정보를 별도로 관리해야 한다.
2) 신고처리기능과 사용자 모니터링
관리자는 사용자로부터 접수된 신고와 문의를 조회하고, 조치 결과와 이력까지 기록할 수 있어야 한다.
또한 사용자명 기반 검색과 지역·상태 기반 검색을 통해 위험 사용자를 빠르게 찾고 가입 승인, 상태 변경, 게시글 숨김 같은 후속 조치를 수행해야 한다.
3) OCR 검수와 운영 규칙 관리
관리자는 OCR 검수 대기열을 조회하고 승인 또는 반려를 결정해야 하며, 검수 결과는 정규화 사전과 추천 규칙 개선에 반영되어야 한다.
추천 태그 조정, 금칙어 관리, 오류·신고 게시글 알림은 품질 관리와 안전 정책을 함께 조정하는 운영 기능으로 본다.
4) 파일럿·공지 운영
관리자는 파일럿 대상 사용자 그룹을 운영하고, 관리자용 푸시·메일 알림과 공지·배너·정책 문구를 발행할 수 있어야 한다.
이 기능들은 관리자 웹 페이지 안에서 운영 이슈를 통합 관리하는 흐름으로 연결되어야 한다.

■ 기능별 간단 설명
1) OCR 추출과 기초 정보 생성
AI Server는 영수증 이미지를 수신해 품목명, 수량, 구매일을 자동 추출하고 품목 후보와 날짜 후보를 초안 형태로 반환해야 한다. 이 과정에서는 OCR 신뢰도와 잘림 여부를 함께 판단하고, 저신뢰 결과는 수동 보정 후보로 표시해야 한다.
2) 정규화와 소비기한 계산
추출된 품목명은 재료 표준명 사전으로 정규화하고, 날짜 후보는 구매일과 유통기한 맥락을 고려해 해석해야 한다.
소비기한 계산은 GPT-4o-mini를 이용해 백그라운드로 수행하며, 결과는 화면에서 바로 확인 가능한 값과 임박도 정보로 연결되어야 한다.
3) 추천과 상세 정보 생성
추천 엔진은 없는 재료를 최대 50%까지 허용하는 추천과 보유 재료만 사용하는 추천을 모두 제공해야 하며, 후보 산출 과정은 보유 재료 일치율과 부족 재료 비율을 함께 고려해야 한다.
추천 결과에는 레시피 상세 정보, 조리순서, 난이도, 예상 조리 시간을 함께 연결해 사용자가 즉시 조리 가능 여부를 판단할 수 있어야 한다.
4) 안전 필터링과 알림
AI Server는 나눔 게시글 작성 시 이름 기반으로 금지 품목을 1차 필터링하고, 위험 식품 후보는 운영 검수 흐름으로 넘겨야 한다. 또한 유통기한 임박 알림을 생성해 소비와 나눔 전환을 유도하고, 추천 결과와 알림 결과를 서비스 API를 통해 안정적으로 전달해야 한다.
5) API 제공과 운영 가능성
AI Server는 영수증 파싱 API, 추천 API, 위험도 평가 API를 제공하고, 레시피 코퍼스 업데이트, 품목 사전 갱신, 오류 로그 수집, 응답시간·오류율 모니터링을 함께 수행해야 한다. 즉 AI Server는 추론 기능뿐 아니라 개선 가능한 운영 시스템으로 설계되어야 한다.
비기능 요구사항
■ 응답시간과 가용성
OCR 초안 생성은 사용자가 대기 가능하다고 느끼는 시간 안에 완료되어야 하며, 지연 시에는 진행 상태를 명확히 안내해야 한다. 운영 환경에서는 OCR 실패율, 검수 대기량, 추천 응답 지연, 신고 급증, 배포 실패를 우선 순위 높은 이벤트로 식별해야 한다.
■ 개인정보와 위치정보 보호
위치 정보와 개인정보는 최소 수집 원칙을 따라야 하며, 세부 좌표와 재식별 가능 문자열은 직접 노출하지 않거나 별도 마스킹 정책으로 관리해야 한다.
파일럿 환경과 운영 환경은 데이터 적재 경로와 알림 채널을 분리해 실험 노이즈가 운영 지표를 오염시키지 않도록 설계해야 한다.
■ 신뢰와 안전 정책
나눔 서비스는 개봉 식품, 상태 확인 불가 식품, 상온 장기 방치 식품을 제한 대상군으로 별도 관리해야 하며, 위험도와 신고 이력을 운영 기준에 반영해야 한다.
문서, 화면, API, 데이터, 운영 대시보드가 같은 용어 체계를 공유하도록 유지하는 것도 비기능 요구사항의 일부로 본다.
데이터 분석 및 전처리
본 장에서는 물무물무 서비스가 사용하는 데이터의 수집 경로와 전처리 기준, 그리고 추천 품질 평가에 활용되는 데이터세트 구성을 정리한다.
분석 대상은 영수증 이미지뿐 아니라 OCR 교정 로그, 레시피 코퍼스, 재료 표준명 사전, 운영 로그를 모두 포함한다. 각 데이터군은 목적이 다르므로 같은 규칙으로 수집하되 다른 평가 기준으로 관리해야 한다.
데이터 수집출처 및 수집방법
■ 원천 데이터군
원천 데이터는 영수증 이미지, OCR 교정 로그, 레시피 코퍼스, 운영 로그로 구분한다. 영수증과 교정 로그는 OCR 품질을 높이기 위한 근거가 되고, 레시피 코퍼스와 운영 로그는 추천 후보 구성과 운영 분석에 사용된다.
수집 방식은 실시간 적재와 주기 수집으로 구분한다. 사용자 입력과 운영 이벤트는 서비스 흐름에서 즉시 적재하고, 공공 API와 실험 코퍼스는 별도 수집 파이프라인을 통해 공통 스키마로 변환한 뒤 저장한다.

데이터 특징추출 및 특징분포
■ 영수증 데이터와 레시피 데이터의 특징
영수증 데이터에서는 품목명, 수량, 구매일, OCR 신뢰도를 추출하고, 레시피 데이터에서는 핵심 재료, 조리시간, 난이도, 카테고리를 구조화한다. 두 데이터는 동일한 재료 체계로 연결되어야 추천 단계에서 함께 사용할 수 있다.

특징 추출 결과는 추천 파생 특징 생성에 직접 사용한다. 보유 재료 일치율, 부족 재료 비율, 임박 재료 포함 여부와 같은 값은 이 단계에서 계산되어 이후 추천 평가의 기준이 된다.
데이터세트 정의 및 생성
■ 데이터세트 생성 원칙
원천 데이터와 운영 로그는 정규화 단계를 거친 뒤 목적별 데이터세트로 분리한다. OCR 검증용 정답셋, 추천 후보 생성을 위한 코퍼스셋, 추천 품질을 확인하는 평가셋, 운영 회귀를 점검하는 피드백셋을 각각 별도로 유지해야 한다.

AI학습모델의 설계
모델 파이프라인의 구성
물무물무의 AI 파이프라인은 영수증 이미지를 곧바로 해석하는 단일 단계 모델이 아니라, OCR 추출과 규칙 기반 정규화, 선택적 생성형 보정 단계를 분리해 운영하는 구조를 전제로 한다.
■ 입력 데이터와 전처리
입력은 사용자 영수증 이미지이며, 전처리 단계에서는 회전 보정, 대비 보정, 잘림 여부 확인, 노이즈 감소를 수행해 OCR 입력 품질을 안정화한다.
■ OCR 추출과 정규화
텍스트 추출은 PaddleOCR을 사용하며, 실제 구현은 `use_angle_cls=True` 설정을 통해 한국어 영수증 회전 보정에 대응한다. OCR 결과는 품목 후보, 수량, 날짜 후보를 포함한 초안으로 생성되고, 이후 FOOD_KEYWORDS와 재료 표준명 사전을 통해 정규화된다.

■ 추천 및 서비스 연계
정규화 결과는 소비기한 계산과 추천 입력으로 연결된다. 추천 단계는 검색 기반 후보 생성과 규칙 기반 재정렬을 우선 적용하고, Qwen은 설명 문구와 대체재 안내를 보정하는 표현 계층으로만 참여한다.
AI학습 알고리즘 선정
■ OCR 엔진 선정
OCR 엔진은 한국어 영수증 대응성과 운영 난이도를 함께 고려해 PaddleOCR을 선택한다. 핵심은 높은 평균 정확도만이 아니라 품목 단위 confidence와 날짜 후보를 안정적으로 분리해 낼 수 있는지에 있다.
■ 규칙 기반 정규화 선정
품목 정규화는 모델 단독 추론보다 규칙 기반 사전 매칭을 우선한다. 브랜드 제거 규칙, 날짜 패턴, 수량·단위 패턴, FOOD_KEYWORDS를 함께 사용하면 서비스에서 설명 가능한 기준으로 결과를 통제할 수 있다.
■ 생성형 모델 활용 범위
Qwen은 저신뢰 영수증 보정과 레시피 설명 문구 정제에만 제한적으로 사용한다. 품목명, 수량, 상태 전이, 우선순위 같은 핵심 구조값은 생성형 모델에 맡기지 않는 것이 운영 안정성과 회귀 통제에 유리하다.

AI학습 파라미터 설정
■ OCR 파라미터
OCR 파라미터의 핵심은 `lang=korean`, `use_angle_cls=True`, `LOW_CONFIDENCE_THRESHOLD=0.80`이다. 이 기준은 전체 문서 점수보다 품목 단위 confidence와 날짜 단위 confidence를 더 중요하게 판단하도록 만드는 운영 설정이다.
■ 정규화 파라미터
정규화 단계는 날짜 패턴, 수량·단위 패턴, 브랜드 제거 토큰, 재료 표준명 사전 일치 여부를 함께 사용한다. 품목 미매칭, 날짜 누락, 수량 누락이 있는 경우에는 `needs_review` 상태로 검수 큐에 적재하는 것이 기본 정책이다.
■ Qwen 보정 파라미터
Qwen 보정 단계는 `temperature=0.15`, `max_tokens=16384`, `QWEN_BASE_URL`, `QWEN_API_KEY`, `QWEN_MODEL` 설정을 기준으로 운용한다. 이 단계의 목적은 창의적 생성이 아니라 저신뢰 결과를 일관된 설명 구조로 정리하는 데 있다.

AI학습 모델의 생성
■ 현재 운영 산출물
현재 MVP 단계에서 생성·배포되는 것은 단일 학습 모델 파일이 아니다. 실제 운영 산출물은 OCR 추출 규칙, 재료 표준명 사전, 위험도 계산 규칙, 추천 재정렬 규칙, Qwen 프롬프트 템플릿의 조합으로 관리된다.
■ 후속 학습 데이터 축적
후속 고도화 단계에서는 OCR 교정 로그와 추천 행동 로그를 정답셋과 평가셋으로 축적해 경량 reranker 또는 규칙 세트 재학습의 기반을 만든다. 그러나 현재 단계의 우선 목표는 end-to-end 학습보다 안정적으로 운영 가능한 추론 파이프라인을 구축하는 데 있다.

그림 5.1 사용자용  메뉴구조도
앱 초기 실행 화면 설계
■ 초기 화면 – 화면번호 initial01
앱을 시작하면 최초로 나타나는 화면이다.
앱 실행 시 로딩 중일 때 나오는 화면이다.
로딩이 완료되면 로그인 화면(initial02)으로 넘어간다.
■ 초기 화면(로그인 화면) - 화면번호 initial02
사용자가 앱을 사용하기 위한 로그인 화면이다.
을 누르면 아이디를 입력할 수 있다.
을 누르면 비밀번호를 입력할 수 있다.
를 누르면 가려진 비밀번호를 볼 수 있다.
올바른 아이디와 비밀번호 입력 후 을 누르면 사용자 정보 수집 화면(initial04-1)으로 이동한다.
■ 회원가입 화면 – 화면번호 initial03
사용자가 서비스 이용을 위해 계정을 생성하는 화면이다.
을 탭하면 이름을 입력할 수 있다.
을 탭하면 사용할 아이디를 입력할 수 있다.
을 탭하면 아이디 중복 여부를 확인할 수 있다.
을 탭하면 비밀번호를 입력할 수 있다.
을 탭하면 비밀번호를 한 번 더 입력할 수 있다. 불일치 시 안내 메시지가 표시된다.
을 탭하면 가려진 비밀번호를 볼 수 있다.
을 탭하면 회원가입이 완료되며 로그인 화면(initial02)으로 넘어간다. 미입력 항목이 있거나 중복확인이 완료되지 않은 경우 진행되지 않는다.
■ 알레르기 정보 입력 화면 – 화면번호 initial04-1
회원가입 완료 후 사용자의 알레르기 정보를 입력하는 화면이다. 해당 정보는 레시피 추천 시 활용된다.
을 선택하면 알레르기 항목이 없는 것으로 설정된다.
계란, 메밀, 땅콩, 대두, 밀, 고등어, 게, 새우, 돼지고기, 우유, 복숭아, 토마토 중 해당하는 항목을 하나 이상 선택할 수 있으며, 중복 선택이 가능하다.
을 탭하면 목록에 없는 알레르기 항목을 직접 입력할 수 있다.
을 탭하면 선택한 알레르기 정보가 저장되며 선호 식재료 선택 화면(intial04-2-1)으로 이동한다.
■선호 식재료 입력 화면(초기 화면) – 화면번호 intial04-2-1
사용자의 선호 식재료를 입력을 받기 위한 화면이다.
안내 문구가 표시된다.
을 탭하면 선호 식재료 입력 없이 다음 화면으로 이동한다.
■ 선호 식재료 입력 화면(검색 화면) - 화면번호 initial04-2-2
에 검색어를 입력하면 연관 식재료 목록이 실시간으로 표시되는 화면이다.
예를 들어 "양"을 입력하면 양고기, 양파 등 입력값과 연관된 식재료 목록이 하단에 표시된다.
을 탭하면 해당 식재료가 선호 식재료로 선택되며 태그 목록에 추가된다.
■선호 식재료 입력 화면 (선택 화면) - 화면번호 initial04-2-3
검색창에서 식재료를 검색하고 선택하면 표시되는 화면이다.
안내 문구 하단에 선택한 식재료가  형태로 표시되며, 각 태그 오른쪽의  를 탭하면 해당 식재료가 목록에서 삭제된다.
을 탭하면 추가로 선호하는 식재료를 검색하여 선택할 수 있으며, 중복 선택이 가능하다.
을 탭하면 선택한 선호 식재료 정보가 저장되며 싫어하는 식재료 입력 화면(initial04-3)으로 이동한다.
■ 기피 식재료 입력 화면 – 화면번호 initial04-3
사용자의 불호 식재료를 입력하는 화면이다. 해당 정보는 레시피 추천 시 활용된다.
을 탭하면 불호하는 식재료를 검색어로 입력할 수 있으며, 입력값에 따라 연관 식재료 목록이 표시된다.
을 탭하면 선택한 기피 식재료 정보가 저장되며 홈 화면(A01)으로 이동한다.
사용자 화면설계(모바일)
■ 하단 네비게이션 바
모든 화면 하단에 고정적으로 표시되는 네비게이션 바로, 주요 메뉴 간 이동을 담당한다.
를 탭하면 내 식자재 화면(A02)으로 이동한다.
을 탭하면 나눔 화면(B02)으로 이동한다.
를 탭하면 레시피 추천 화면(C01)으로 이동한다.
를 탭하면 1:1 채팅 목록화면(D01)으로 이동한다.
를 탭하면 내 정보 화면(E01)으로 이동한다.
■ 내 식자재 화면(등록 전) – 화면번호 A01
등록된 식자재가 없을 때 표시되는 화면이다.
화면 중앙에 "등록된 재료가 없어요. '추가' 버튼을 눌러 재료를 추가해 주세요." 안내 문구가 표시된다.
을 탭하면 식자재를 추가 방법을 선택하는 화면(A02-1)으로 이동한다.
■ 내 식자재 화면(등록 후) - 화면번호 A02
사용자의 냉장고에 등록된 식자재 목록을 확인하는 화면이다.
소비기한 임박 알림 배너는 화면 상단에 표시되며, 소비기한이 3일 이내로 임박한 식자재의 개수를 안내한다.
을 탭하면 등록된 식자재를 검색어로 조회할 수 있다.
<식자재 목록>은 등록된 식자재를 리스트 형태로 표시하며, 각 항목은
- 식자재의 분류 이미지
- 식자재 이름
- 남은 소비기한 (D-day 형태로 표시, 예: D-1)
- 소비기한 날짜
- 사용·미사용·사용중 상태 표시
소비기한이 3일 이내인 식자재의 경우 남은 소비기한이 와 같이 해당 항목이 빨간색으로 강조 표시된다.
을 탭하면 해당 식자재의 수정 화면(A02-3)으로 이동한다.
을 탭하면 식자재를 추가 방법을 선택하는 화면(A02-1)으로 이동한다.
■ 식자재 등록 방식 선택 화면 -  화면번호 A02-1
내 식자재 화면에서 을 탭하면 식자재 등록
방식을 선택할 수 있는 화면이 표시된다.
을 탭하면 식자재 정보를 사용자가 직접 입력하는 화면(A02-1-4-1)으로 이동한다.
을 탭하면 갤러리에서 영수증 이미지를 선택하여 식자재를 등록하는 화면(A02-1-2)으로 이동한다.
을 탭하면 카메라가 실행되어 영수증을 직접 촬영하여 식자재를 등록하는 화면(A02-1-1)으로 이동한다.
■ 영수증 촬영 화면 – 화면번호 A02-1-1
식자재 추가 선택 화면(A02-1)에서 을 탭하면 카메라가 실행되는 화면이다.
<카메라 화면>에서 영수증을 카메라 가이드 영역에 맞춰 촬영할 수 있다.
을 탭하면 영수증 사진이 촬영된다. 이후 영수증 분석 로딩 화면(A02-1-3)으로 이동한다..
■ 영수증 갤러리 등록 화면 – 화면번호 A02-1-2
식자재 추가 선택 화면(A02-1)에서 을 탭하면 사용자의 앨범(갤러리)이 실행되는 화면이다.
<앨범 화면>에서 등록할 영수증 이미지를 선택할 수 있다.
이미지를 선택하면 영수증 분석 로딩 화면(A02-1-3)으로 이동한다.
■ 영수증 분석 로딩 화면 – 화면번호 A02-1-3
영수증 촬영 또는 등록 완료 후 영수증을 분석하는 동안 표시되는 화면이다.
안내 문구가 표시되며, 분석이 완료될 때까지 해당 화면이 유지된다.
분석이 완료되면 식자재 수정/추가 화면(A02-1-4-3)으로 이동한다.
■ 식자재 직접 입력 화면 – 화면번호 A02-1-4-1
A02-1 화면에서 직접 입력을 탭 했을 때 표시되는 화면이다.
화면 중앙에 “'추가' 버튼을 눌러 재료를 추가해 주세요." 안내 문구가 표시된다.
을 탭하면 식자재 추가 창(A02-1-4-2) 화면으로 이동한다.
■ 식자재 직접 입력 화면 – 화면번호 A02-1-4-2
식자재를 직접 추가할 수 있는 화면이다.
을 탭하면 등록할 식자재 이름을 직접 입력할 수 있다.
을 탭하면 해당 식자재의 유통기한을 입력할 수 있다.
을 탭하면 식자재의 카테고리를 선택할 수 있다.
을 탭하면 추가로 식재료를 추가할 수 있는 창이 뜬다.
모든 정보를 입력하고 완료를 누르면 홈화면(A02)으로 이동한다.
■ 식자재 입력/수정 화면 – 화면번호 A02-1-4-3
영수증 분석 후 나오는 화면이다.
영수증 분석 결과가 창으로 표시되며, 추가적으로 수정을 하거나 다른 식재료를 추가하는 것이 가능하다.
모든 정보를 수정/추가 완료 후 을 탭하면 홈화면(A02)으로 이동한다.
■ 동네 설정 화면 – 화면번호 B01
나눔 탭에서 동네가 설정되지 않은 경우 표시되는 화면이다.
화면 중앙에  안내 문구가 표시된다.
을 탭하면 사용자의 동네를 검색하고 설정할 수 있는 화면(B01-1)으로 이동한다.
■ 동네 검색 화면 – 화면번호 B01-1
동네를 검색하여 설정할 수 있는 화면이다.
을 탭하면 동네 이름을 직접 입력하여 검색할 수 있다. 입력값에 따라 연관 동네 목록이 표시된다.
을 탭하면 기기의 GPS를 기반으로 사용자의 현재 위치를 자동으로 인식하여 동네가 설정된다.
■ 나눔 화면 - 화면번호 B02
사용자 주변 지역의 식자재 나눔 게시글 목록을 확인하는 화면이다.
지역은 화면 상단에 표시되며, 사용자의 현재 위치를 기반으로 "~동" 형태로 현재 설정된 지역명이 표시된다.
나눔 목록은 해당 지역의 나눔 게시글을 리스트 형태로 표시하며, 각 항목은 아래 정보로 구성된다.
- 식자재 이미지
- 식자재 이름
- 식자재 상세설명
게시글 항목을 탭하면 해당 나눔 게시글의 상세 화면(B02-2)으로 이동한다.
■ 나눔 등록 선택 팝업 – 화면번호 B02-1
나눔 화면(B02)에서 을 탭하면 나눔 게시글 작성 방식을 선택할 수 있는 화면이 표시된다.
를 탭하면 소비기한이 임박한 식자재 목록이 팝업(B02-1-1)으로 표시되며, 나눔할 식자재를 선택할 수 있다.
을 탭하면 나눔할 식자재 정보를 직접 입력하여 게시글을 작성하는 화면(B02-1-2)으로 이동한다.
■ 임박 식재료 팝업 화면 – 화면번호 B02-1-1
나눔 글쓰기 화면에서 를 탭하면 소비기한이 임박한 식자재 목록이 팝업으로 표시된다.
<임박 식재료 목록>은 소비기한이 짧은 순서대로 리스트 형태로 표시되며, 각 항목은 아래 정보로 구성된다.
- 소비기한 (예: D-1, D-2)
- 식자재 이름 (예: 감자, 시금치)
식자재 항목을 탭하면 해당 식자재가 나눔 게시글 작성 화면(B02-1-2)으로 이동한다.
■ 나눔 게시글 작성 화면- 화면번호 B02-1-2
나눔할 식자재 정보를 입력하여 게시글을 작성하는 화면이다.
을 탭하면 갤러리에서 나눔할 식자재 사진을 선택하여 등록할 수 있다.
을 탭하면 나눔 게시글의 제목을 입력할 수 있다.
을 탭하면 나눔할 식자재에 대한 상세 설명을 입력할 수 있다.
으로 나눔할 식자재의 분류를 선택할 수 있다.
화면은 미개봉 가공식품 또는 건강기능식품을 선택하면 나오는 화면이다. 을 탭하면 나눔할 식자재의 유통기한을 입력할 수 있다.
을 탭하면 입력한 정보를 바탕으로 나눔 게시글이 등록되며 나눔 화면(B02)으로 이동한다. 미입력 항목이 있을 경우 진행되지 않는다.
■ 나눔 게시글 상세 화면 – 화면번호 B02-2
나눔 목록(B02)에서 게시글을 탭하면 해당 나눔 게시글의 상세 정보를 확인하는 화면이다.
식자재 사진이 화면 상단에 크게 표시된다.
작성자 정보는 사진 하단에 표시되며, 작성자의 닉네임과 거주 지역(~동)이 함께 표시된다.
식자재 이름은 작성자 정보 하단에 표시된다.
상세 설명은 식자재 이름 하단에 표시되며, 작성자가 입력한 식자재에 대한 상세 설명이 표시된다.
을 탭하면 해당 게시글 작성자와 채팅 화면(D01-1)으로 이동한다.
■ 레시피 메인 화면 - 화면번호 C01
사용자가 레시피와 레시피 추천을 받을 수 있는 화면이다.
을 탭하면 레시피 내용 화면(C01-1)로 이동한다.
을 탭하면 레시피 추천받기 화면(C01-2)로 이동한다.
을 누르면 원하는 레시피를 검색할 수 있다,

프로세스(기능) 설계
시스템 아키텍처
그림 6.1 물무물무 시스템 아키텍처
그림 6.1은 모바일 앱, 관리자 웹, Spring Boot API, FastAPI 기반 OCR·추천 서버, MariaDB, 객체 스토리지가 하나의 서비스 흐름으로 연결되는 구조를 보여준다.
사용자와 운영자는 서로 다른 화면을 사용하지만, 모든 요청은 공통 API와 운영 로그 체계로 수렴하도록 설계해 상태 추적과 대응 기준이 분리되지 않게 했다.
■ 서비스 계층 및 로드 밸런싱 구조
본 시스템은 사용자용 React Native 앱과 관리자용 React 웹을 분리해 각 채널의 사용 맥락에 맞는 인터페이스를 제공한다. 외부 요청은 먼저 Nginx 로드 밸런서를 통과하며, 이 계층에서 정적 자산 전달, SSL 종료, API 라우팅, 장애 우회가 함께 수행된다.
로드 밸런서는 Spring Boot 애플리케이션 서버로 트래픽을 분산하고, 특정 인스턴스 장애 시에도 다른 인스턴스로 요청을 우회해 서비스 연속성을 유지한다.
■ 마이크로서비스 기반의 서버 구성
일반적인 비즈니스 로직과 영속성 처리는 Spring Boot가 담당하고, 영수증 인식과 추천 계산처럼 Python 생태계가 유리한 영역은 FastAPI 기반 AI 서버로 분리한다.
이 구조는 AI 연산 부하가 본 서비스 응답 시간에 직접 전파되는 것을 줄이고, OCR·추천 모듈을 별도 주기로 배포·확장할 수 있게 한다.
■ 데이터 관리 및 스토리지 전략
정형 데이터는 MariaDB에 저장해 사용자, 재고, 나눔, 알림, 운영 로그의 관계를 일관되게 관리한다. 영수증 이미지와 게시글 첨부 파일 같은 비정형 데이터는 객체 스토리지에 보관해 애플리케이션 서버의 저장소 부담을 낮춘다.
데이터베이스와 스토리지는 각각의 용도에 맞게 분리해 조회 성능과 보존 안정성을 동시에 확보하고, 운영자는 로그와 메타데이터를 통해 파일과 레코드의 연결 상태를 추적할 수 있다.
■ CI/CD 자동화와 배포 운영
배포는 새 버전을 충분히 기동한 뒤 트래픽을 전환하는 방식으로 설계해, 사용자 서비스의 가용성을 떨어뜨리지 않고 기능을 교체할 수 있게 한다.
GitHub Actions와 Docker를 연동한 CI/CD 파이프라인은 빌드, 테스트, 이미지 생성, 환경 배포를 자동화한다. 운영 환경 반영은 서비스 API, AI 서버, 저장소 구성을 단계적으로 갱신하는 방식으로 관리하며, 실패 시 즉시 이전 운영 상태로 되돌릴 수 있는 기준을 함께 유지한다.
사용자 기능설계 (Sequence Diagram)
사용자 기능 설계는 chapter 2에서 정의한 사용자 요구기능을 실제 서비스 흐름 단위로 재구성한 것이다.
각 소절은 기능 하나당 하나의 시퀀스 다이어그램을 대응시키고, 정상 흐름과 예외 흐름, 연동 데이터와 운영 포인트를 함께 읽을 수 있도록 구성한다.
특히 인증, 영수증 OCR, 소비기한 계산, 추천, 나눔과 채팅 흐름은 화면 이동 자체보다 상태 전이와 후속 행동 연결을 중심으로 설명한다.
1) 소셜 로그인
기능 목적: 사용자가 카카오 계정으로 빠르게 서비스에 진입할 수 있도록 한다.
선행 조건: 사용자가 카카오 인증을 완료할 수 있어야 하며, 서비스는 OAuth 연동 정보를 수신할 수 있어야 한다.
그림 6.2.1 소셜 로그인 시퀀스 다이어그램
■ 정상 흐름
- 사용자가 소셜 로그인 버튼을 선택한다.
- 앱이 카카오 인증 화면으로 이동한다.
- 인증 성공 시 서비스 API가 사용자 식별 정보를 확인한다.
- 기존 회원이면 로그인 세션을 발급하고, 신규 사용자는 가입 흐름으로 연결한다.
■ 예외 흐름
- 카카오 인증이 취소되면 로그인 화면으로 복귀한다.
- OAuth 토큰 검증에 실패하면 오류 메시지를 표시한다.
- 기존 계정과 소셜 계정 연결이 충돌하면 수동 확인 절차로 전환한다.
■ 연동 데이터/API
- OAuth 인증 API
- 사용자 계정 테이블
- 세션 토큰 발급 API
2) 일반 로그인
기능 목적: 아이디와 비밀번호 기반의 기본 로그인 절차를 제공한다.
선행 조건: 사용자가 사전에 일반 회원가입을 완료했어야 한다.
그림 6.2.2 일반 로그인 시퀀스 다이어그램
■ 정상 흐름
- 사용자가 아이디와 비밀번호를 입력한다.
- 서비스 API가 계정 상태를 조회한다.
- 인증이 성공하면 세션을 발급하고 메인 화면으로 이동한다.
- 로그인 이력은 운영 로그에 기록한다.
■ 예외 흐름
- 비밀번호가 일치하지 않으면 재입력을 요청한다.
- 계정이 비활성 상태이면 안내 메시지를 출력한다.
- 연속 실패 시 추가 인증 또는 잠금 정책을 적용한다.
■ 연동 데이터/API
- 인증 API
- 사용자 계정 테이블
- 로그인 이력 로그
3) 소셜 로그인 기반 회원가입
기능 목적: 카카오 계정을 사용해 간단하게 회원가입을 완료하도록 한다.
선행 조건: 카카오 인증이 가능해야 하며, 사용자는 최초 가입 상태여야 한다.
그림 6.2.3 소셜 로그인 기반 회원가입 시퀀스 다이어그램

- 필수 약관 동의가 없으면 다음 단계로 이동하지 않는다.
- 이미 가입된 계정이면 로그인 흐름으로 전환한다.
■ 연동 데이터/API
- OAuth API
- 사용자 프로필 테이블

관리자 기능설계 (Sequence Diagram)
관리자 기능 설계는 chapter 2 관리자 기능표를 기준으로 운영 승인, 검수, 공지, 파일럿 관리, 권한 조정 흐름을 실제 처리 단위로 재구성한 것이다.
각 소절은 운영자의 판단, 시스템 반영, 감사 로그, 사용자 영향 범위를 함께 읽을 수 있게 구성한다.
1) 관리자 웹 대시보드 조회
기능 목적: 관리자는 운영 현황, 처리 대기, 주요 알림을 한 화면에서 확인해야 한다.
선행 조건: 관리자 계정으로 로그인되어 있어야 하며, 대시보드 접근 권한이 부여되어야 한다.
그림 6.3.1 관리자 웹 대시보드 조회 시퀀스 다이어그램
■ 정상 흐름
- 관리자가 관리자 웹에 접속한다.

■ 연동 데이터/API
- 알림 API
- 메일 API
- 사용자 수신 설정 API
■ 운영 포인트

15) 환경설정 관리

AI학습서버 기능설계
AI학습서버 기능 설계는 모델 내부 이론보다 서비스 처리 흐름을 설명하는 데 목적이 있다.
OCR, 정규화, 소비기한 계산, 추천, 안전 필터링, API 제공, 운영 모니터링까지를 하나의 서비스 체계로 본다.
1) OCR 수집모듈
기능 목적: 영수증 이미지를 서비스 입력으로 수신하고 OCR 처리의 시작점을 만든다.
선행 조건: 사용자가 영수증 이미지를 업로드할 수 있어야 하며, 이미지 스토리지와 서비스 API가 연결되어 있어야 한다.
그림 6.4.1 OCR 수집모듈 시퀀스 다이어그램
■ 정상 흐름
- 사용자가 영수증 이미지를 업로드한다.
- 서비스 API가 이미지 메타데이터를 검증한다.
- OCR 서버로 분석 요청이 전달된다.
- 수신 결과는 추적용 trace_id와 함께 기록된다.
■ 예외 흐름
- 업로드 실패 시 재시도를 안내한다.
- 이미지 형식이 맞지 않으면 입력을 차단한다.
- 파일 크기 제한 초과 시 축소 또는 재등록 흐름으로 전환한다.
■ 연동 데이터/API
- 이미지 업로드 API
- 이미지 스토리지
- OCR 요청 로그
■ 운영 포인트
- 업로드 실패율과 재시도율을 분리해서 집계한다.
- 비정상 이미지 유형은 운영 로그로 남긴다.
2) OCR 텍스트 추출
기능 목적: 영수증 이미지에서 식별 가능한 텍스트를 추출한다.
선행 조건: OCR 서버가 이미지 파일과 추적 정보를 수신해야 한다.
그림 6.4.2 OCR 텍스트 추출 시퀀스 다이어그램
■ 정상 흐름
- OCR 서버가 라인 단위 텍스트를 추출한다.
- 품목 후보와 수량 후보를 분리한다.
- 추출 결과를 구조화 초안으로 반환한다.
- 초안은 후속 정규화 단계로 넘겨진다.
■ 예외 흐름
- 글자 인식률이 낮으면 저신뢰 상태를 반환한다.
- 일부 라인이 잘리면 누락 플래그를 남긴다.
- OCR 응답이 지연되면 검수 대기 상태로 전환한다.
■ 연동 데이터/API
- OCR 추출 API
- OCR 결과 테이블
- 검수 큐
■ 운영 포인트
- confidence 분포를 주기적으로 확인한다.
- 저신뢰 샘플은 재학습/규칙 보정 후보로 저장한다.
3) 품목명 자동 추출
기능 목적: OCR 결과에서 품목명을 식재료 표준명 기준으로 식별한다.
선행 조건: OCR 초안이 존재해야 한다.

- 유사 후보를 포함해 품목명을 정규화한다.
- 정규화 결과를 재고 초안에 반영한다.
■ 예외 흐름
- 후보가 여러 개면 검수 대상으로 남긴다.
- 표준명 사전에 없는 품목은 임시 코드로 저장한다.
- 식별 실패 시 수동 입력 경로를 유지한다.
■ 연동 데이터/API
- 식재료 표준명 사전
- 품목 정규화 API
- 재고 초안 테이블
■ 운영 포인트
- 표준명 매핑 실패율을 관리한다.
- 동의어 사전 갱신 이력을 남긴다.
4) 수량 및 구매일 자동 추출
기능 목적: 수량과 구매일을 함께 추출해 소비기한 계산의 기초값을 만든다.

- 소비기한 계산 모듈로 전달한다.
■ 예외 흐름
- 수량 단위가 불명확하면 수동 보정 후보로 둔다.
- 구매일이 누락되면 기본 규칙을 적용한다.
- 형식 충돌이 발생하면 검수 큐로 넘긴다.

- 소비기한 계산 모듈

- 단위별 오인식 패턴을 분류한다.

기능 목적: 정규화 실패 항목에 대해 수동 보정 가능한 후보를 반환한다.

API 설계
서버/클라이언트 구조
클라이언트는 사용자 앱과 관리자 웹으로 분리되며 두 채널 모두 동일한 도메인 API를 사용하지만 권한 범위는 다르게 제한된다.
서비스 API 서버는 인증, 재고, 나눔, 알림, 운영 기능을 담당하고, OCR과 추천 계산은 별도의 AI 서버로 위임해 책임 경계를 분명히 한다.
정적 이미지와 영수증 원본은 스토리지에 저장하고, DB에는 메타데이터와 상태 정보만 보관한다.
배포 구조는 컨테이너 단위로 모듈을 분리해 확장과 장애 격리를 쉽게 만든다.
그림 7.1 서버 인터페이스 구조
클라이언트는 React Native 앱과 관리자 웹으로 나뉘며, 서비스 API 서버가 인증·재고·나눔·알림을 담당한다.
OCR/추천 모듈은 FastAPI 기반 AI 서버로 분리해 호출하며, 영수증 이미지와 정적 리소스는 스토리지에 저장한다.
REST API 정의 (사용자)
사용자 REST API는 인증, 위치 검증, 영수증·재고, 추천, 나눔, 채팅, 알림 기능을 기준으로 정의한다.
■ 사용자 API 정의
사용자 API는 모바일 앱에서 직접 호출되는 엔드포인트를 대상으로 하며, 요청과 응답은 JSON 형식으로 통일한다.
인증이 필요한 요청은 Authorization 헤더를 사용하고, 쓰기 요청은 멱등 키를 적용할 수 있다.
GET API는 조회, POST API는 생성, PATCH API는 상태 갱신, DELETE API는 삭제·철회 성격으로 정리한다.
동네 인증, OCR 초안 확정, 추천 숨김, 회원 탈퇴처럼 사용자 행동 흐름을 바꾸는 요청은 결과 코드와 후속 상태를 함께 반환한다.
대표 사용자 API는 로그인, 회원가입, 동네 인증, 영수증 등록, 재고 조회·수정, 추천, 나눔 게시글, 알림, 회원 탈퇴 기능을 포함한다.

관리자 REST API는 인증, 대시보드, 신고 처리, OCR 검수, 게시글 운영, 파일럿 운영 기능을 기준으로 정의한다.
■ 관리자 API 정의
관리자 API는 운영 웹에서 호출되며, 모든 상태 변경 요청은 사유와 작업자 식별자를 함께 기록하는 것을 기본 원칙으로 한다.
GET API는 모니터링과 탐색, POST API는 메모·공지·재처리 요청, PATCH API는 상태 전이, DELETE API는 철회·정리 요청으로 구분한다.

REST API 정의 (AI Server)
AI Server API는 OCR 전처리, OCR 텍스트 추출, 날짜 정규화, 품목 정규화, 위험도 계산, 추천 생성 기능을 기준으로 정의한다.
■ AI Server API 정의
AI Server는 서비스 API가 내부적으로 호출하는 계산 계층이며, 응답은 result_code, trace_id, model_version, confidence를 함께 반환하는 것을 기본 원칙으로 한다.
GET API는 품질·버전·작업 조회, POST API는 OCR·추천 계산 요청, PATCH API는 운영 파라미터 갱신, DELETE API는 캐시·배치 정리 요청으로 구분한다.

만료 작업 정리
대표 AI Server API는 전처리, OCR 라인 추출, 날짜 정규화, 품목 정규화, 위험도 계산, 추천 후보 생성, 품질 지표 조회로 구성한다.
아래 표는 Method 기준으로 AI Server 엔드포인트를 묶어 서비스 API가 어떤 계산 모듈을 호출하는지 빠르게 파악할 수 있게 정리한 것이다.

데이터 설계
ERD
■ 사용자 및 선호 데이터
User는 서비스 사용자 계정의 기준 정보를 관리하며, Untitled는 사용자별 선호 및 기피 식재료를 저장하는 보조 테이블로 사용된다.
■ 재고 및 나눔 데이터
Groceries는 냉장고 재고를 저장하고, Share와 Share_Picture는 재고 중 나눔 가능한 식재료를 게시글과 이미지 단위로 전환 관리한다.
■ 채팅 및 신고 데이터
ChatRoom, ChatRoomParticipant, ChatMessage는 게시글 기반 채팅 흐름을 저장하고, Report는 신고 이력을 기록한다.
■ 레시피 데이터
Recipe, Recipe_Ingeredient, Recipe_Step은 추천 서비스의 참조 데이터로 사용되며 기본정보, 재료, 조리 단계를 분리해 저장한다.
■ 핵심 관계 해석
User는 Groceries, Share, ChatMessage, Report의 중심 엔터티이며, Share는 ChatRoom 및 Share_Picture와 연결된다.
users와 inventory_items는 1:N 관계이며, receipt_uploads는 inventory_items 생성의 근거 기록으로 연결된다.
inventory_items는 notifications, recommendation_logs, share_posts와 모두 연결되어 행동 전환 흐름을 추적한다.
share_posts는 share_chats와 1:N 관계이고, 완료 이벤트가 inventory_items 상태 변화와 연결된다.
review_logs는 receipt_uploads, inventory_items, recommendation_logs와 연결되어 검수 결과를 재현 가능하게 만든다.
핵심 엔터티는 사용자, 재고품목, 영수증, 레시피, 나눔게시글, 채팅메시지, 알림, 검수로그이다.
ERD에는 각 엔터티의 PK/FK와 사용자-재고-나눔-알림 간 관계를 표시한다.
논리적설계(테이블명세서)
■ 설계 범위
본 절은 ERD에 포함된 핵심 테이블을 기준으로 사용자, 재고, 나눔, 채팅, 신고, 레시피 도메인 순으로 명세를 정리한다.
8.2는 안전한 16행 donor 양식을 재사용해 테이블 정의를 통일된 형식으로 기술한다.

소스디렉터리 구조
소스디렉터리 구조는 Spring MVC 패턴을 기준으로 View, Controller, Service, Repository 계층을 분리하고, AI/OCR 모듈은 FastAPI 기반 별도 패키지로 운영하는 방식으로 구성한다.
웹 템플릿과 정적 자원은 resources 하위에 두고, 서비스 API는 Java 계층, AI 분석 로직은 Python 패키지로 분리해 변경 범위와 배포 단위를 명확히 나눈다.

명칭표준안(변수, 함수, 파일명 등)

서비스 계층과 DB 물리명 사이의 차이가 있더라도, 설계서에서는 어떤 규칙을 우선 적용하는지 명확하게 적어 두는 것이 좋다.
